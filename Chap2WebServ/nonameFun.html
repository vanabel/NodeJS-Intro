第二节. 回调函数与事件驱动

还记得上次说到createServer的参数是一个函数吗？而且该函数还没有名字！我们先来看看“正常”的代码应该是怎样的。

// file: index.js
var http=require('http');

function onRequest(req, res){
    console.log('Request received')
    res.writeHead(200, {'content-type': 'text/html; charset=utf-8'});
    res.write('世界，你好!');
    res.end();
}

http.createServer(onRequest).listen(8000);
console.log('server started at http://localhost:8000');

自己试验下我们的代码还能够运行吗？

事件驱动

回顾熟悉的PHP+nginx架构，当服务器收到请求时，nginx就新开一个进程，执行对应的PHP脚本，并将结果反馈给请求. 但是，对Node.js问题来了，因为Node.js是单线程的. 那么这时如何处理请求呢？试想请求可以在任何时刻到来，我们怎么在不开新进程的前提下处理他们呢？这涉及到两个非常关键的概念:事件驱动与非阻塞操作.

再想一遍PHP+nginx处理请求的过程，你会发现开新进程其实不是非常必要。因为你可以将这一过程放到一个函数中(原来是PHP脚本)，函数执行完毕可以返回值作为请求的应答。是不是很绝妙？此外什么时候触发该函数和原来一样：当有请求这个事件发生时，就调用该函数。由此可见，该函数不需要给一个名字，因为我们准确知道什么时候调用该函数---其实就是事件发生时，故我们将其绑到一起。由此，也不难理解将其称为回调函数的含义了：

当事件发生时我们调用该函数，并将函数执行的结果返回给服务器作为该请求的回应。

具体到上面的例子，当服务器没有收到来自浏览器的请求时，回调函数onRequest其实并没有调用，这时控制台输出

server started at http://localhost:8000

一旦我们访问http://localhost:8000, 则该回调函数就被调用了，此时控制台输出

Request received

细心的你也许发现有两个事件发生了，具体是哪两个事件我们将在下节看到。
